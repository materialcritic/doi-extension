# DOI Grabber — Build Log

A Chrome extension (Manifest V3) that detects the DOI of the academic paper you're viewing, checks whether it's available on Sci-Hub, and can download it via a local Python script (`scihub_download.py`) through a Native Messaging host (`doi_host.py`).

## Architecture

- **`extension/`** — the Chrome extension (loaded unpacked from `~/Downloads/doi-extension/extension`)
  - `manifest.json` — MV3 manifest (permissions: `activeTab`, `scripting`, `nativeMessaging`, `storage`, `notifications`, `alarms`, `contextMenus`)
  - `content.js` — scans the page for a DOI (URL, meta tags, page text) and for title/author metadata; auto-runs on every page load; wrapped in a `window.__doiGrabberLoaded` guard so double-injection (manifest auto-inject + popup.js's fallback `executeScript`) doesn't throw on re-declared top-level `const`s
  - `background.js` — service worker; runs the availability check queue, badges the toolbar icon, relays native-messaging traffic, fires the "unavailable" notification, runs the journal/author watchlist alarms, handles the right-click "Grab DOI from selection" context menu
  - `popup.html` / `popup.js` — the toolbar popup UI
  - `options.html` / `options.js` — settings page (output folder, python path, script path, mirror list, color theme, journal/author watchlists)
  - `theme.js` — shared color-theme module (dark + 4 light palettes: parchment/slate/sage/minimal) loaded by every page; hides `<html>` until `chrome.storage.sync`'s `theme` resolves to avoid a flash, exposes `window.toggleTheme()`/`window.setTheme()`
  - `author.html` / `author.js` — "Download All Works" page for an author (Crossref `query.author` search)
  - `collaborators.html` / `collaborators.js` — "Common Collaborators" page, tallies co-author frequency from the same author search
  - `issue.html` / `issue.js` — "Download This Issue" page, lists every article in a paper's volume/issue via Crossref's `/journals/{issn}/works`
  - `search.html` / `search.js` — "Similar Papers" results page, driven by a crude keyword-extraction-from-abstract → Crossref `query.bibliographic` search; results can themselves chain into further searches
  - `vendor/qrcode.js` — vendored, fully-offline QR encoder (public-domain `qrcode-generator` port)
  - `icons/` — pirate skull-and-cutlasses icon (recreated as SVG→PNG since the original pasted image couldn't be saved directly)

- **`native-host/`** (lives at `~/doi-extension/native-host`, moved out of Downloads early on due to macOS Gatekeeper/quarantine issues blocking execution from a Downloads-derived path)
  - `doi_host.py` — the native messaging host; spawns `scihub_download.py`, streams progress back to the extension, handles `reveal` (Finder) and `check` (availability) actions
  - `com.doi_grabber.host.json` — the native messaging manifest registered in Chrome's `NativeMessagingHosts` directory
  - `download_log.txt` — append-only log of every download attempt (timestamp, status, DOI, filepath/size or error)
  - `mirror_health.json` — per-mirror failure tracking (skips a mirror for 30 min after 3 failures)

- **`/Users/floppa/bin/scihub_download.py`** — the pre-existing CLI script this whole extension wraps. Downloads from Sci-Hub mirrors, saves to `~/Downloads/autorename` by default.

## What got built, roughly in order

1. **Initial scaffold** — MV3 extension + native messaging host + installer script, wired to the existing `scihub_download.py`.
2. **Fixed "native host has exited"** — root cause was the `com.apple.quarantine` xattr on `doi_host.py` (added when saved via TextEdit from a Downloads-derived path), which macOS Gatekeeper silently blocked Chrome from executing.
3. **Fixed the actual download failure** — Chrome launches the host with a different `python3` (system one, no `requests` installed) than the terminal's Homebrew one; hardcoded `PYTHON_BIN` to the correct interpreter.
4. **Live progress streaming** — switched `doi_host.py` from `subprocess.run` (blocking, all-or-nothing) to `subprocess.Popen` with line-by-line streaming, plus a `RESULT:` JSON marker line in `scihub_download.py` for the host to parse the final outcome. Popup shows a live scrollable log.
5. **Dark theme** for the popup.
6. **Custom icon** — recreated the user's pasted pirate skull/cutlasses image as SVG, rendered to 16/48/128px PNGs (via a throwaway `cairosvg` venv).
7. **Settings page** — output folder, python interpreter path, script path, and Sci-Hub mirror list, stored via `chrome.storage.sync` and passed through with every native-messaging request instead of being hardcoded.
8. **Availability badge** — `content.js` auto-detects the DOI on every page load and messages `background.js`, which runs a `--check`-only pass (added to `scihub_download.py`) and badges the toolbar icon: green ✓ if a PDF is genuinely found, red ✗ if confirmed not found, no badge if inconclusive (error/timeout).
   - Fixed: checks were running concurrently across tabs, which caused Sci-Hub mirrors to flake under load — serialized into a one-at-a-time queue.
   - Fixed: badge was being cleared by spurious `"loading"` tab-status events without an actual navigation — now only clears on a real URL change (`changeInfo.url`).
9. **Reveal in Finder** — after a successful download, a button that asks the native host to run `open -R <path>`.
10. **Right-click "Grab DOI" context menu** — added, then **reverted** at the user's request after it didn't work reliably.
11. **Copy Sci-Hub Link** — reuses the `--check` mode's `pdf_url` field; copies the resolved mirror URL to the clipboard without downloading.
12. **"Search Google Instead" fallback** — when a paper is unavailable, a button opens a Google search for `<title> <author>` (extracted from page meta tags, with cleanup for sites like PhilPapers where `document.title` is formatted `"Author, Title - SiteName"` and would otherwise duplicate the author in the query).
13. **Desktop notification on unavailable** — when the background check confirms a paper isn't on Sci-Hub, a native notification pops up automatically with a "Search Google" button, instead of requiring the user to open the popup first.
    - Debugged via a manual `osascript` test notification to confirm the issue was Chrome's OS-level notification permission (System Settings → Notifications → Google Chrome), not the extension.
14. **"More by This Author"** button — opens a Google Scholar search for the detected author.
15. **Made "Search Google Instead" always visible** (previously only shown conditionally on failure).
16. **Parallel mirror racing** — `get_pdf_url` now fires requests to all Sci-Hub mirrors simultaneously via daemon threads + a queue (deliberately *not* `ThreadPoolExecutor`, since `concurrent.futures` joins all its worker threads at interpreter exit regardless of `shutdown(wait=False)`, which would silently cancel the speed win) and returns as soon as the first one succeeds.
17. **Mirror health tracking** — `mirror_health.json` records failures per mirror; a mirror failing 3+ times gets skipped for 30 minutes (unless that would skip every mirror).
18. **Bug fixes from real-world use:**
    - Renamed "Run Script" → "Download".
    - **Reveal in Finder kept failing** because a separate, pre-existing local tool — `~/Scripts/auto_rename_by_doi.py`, triggered by a macOS Folder Action on the output folder — renames downloaded PDFs based on a Crossref DOI lookup shortly after they land. Fixed by:
      - Reading `rename_log.csv` (which that tool writes) to resolve the new filename, instead of `os.listdir()` on the folder (which hit a macOS TCC "Operation not permitted" error specific to *listing* a Downloads subfolder).
      - Discovering the renamer is Folder-Action-triggered with a live Crossref API call (up to 15s), not an instant watcher — extended the poll/retry window from 3s to 20s, with a "Locating file…" status message so the popup doesn't look frozen.
    - **False "unavailable" badge/notification** — Sci-Hub mirrors are flaky enough that a single mirror-race can come back empty even when the paper is genuinely available moments later. Added a retry: if every mirror fails, wait 2s and race again before concluding "unavailable."

## Second build session — features added, roughly in order

19. **Keyboard shortcuts** — `Alt+D` (Option+D) downloads the current paper if available; `Alt+F` (Option+F) searches Google for it, and works even with no DOI detected at all (re-scans the page for title/author on every press). Settings page shows the current bindings read-only (Chrome doesn't allow an extension to set its own shortcuts) with a button linking to `chrome://extensions/shortcuts`.
20. **Settings page redesign** — modern centered card layout with CSS variables, replacing the old left-aligned serif page.
21. **Mirror Health panel** (options page) — per-mirror fail count, cooldown countdown, and latency (`last_latency_ms`, timed in `scihub_download.py`'s mirror race). Stale entries (untouched 4+ days, tracked via `last_seen`) are pruned on save and filtered from the display.
22. **Download Stats panel** (options page) — total-ever / last-7-weeks / last-7-months / last-year counts, parsed from `download_log.txt`.
23. **Corrupt-file detection** — after download, `scihub_download.py` checks the `%PDF-` magic header; a mismatch (mirror served an HTML error page) logs `CORRUPT` and the popup shows a red "Delete Corrupt File" button (native host `delete_file` action).
24. **Availability accuracy fix** — `get_pdf_url`'s retry count raised from 2→4 attempts (3s apart) after a real false-negative case (paper confirmed available on Sci-Hub but the extension badged it unavailable).
25. **Mirror cooldown shortened** 30→10 minutes.
26. **QR Code button** — encodes the current Sci-Hub article link (not just the DOI) using a vendored, fully-offline QR encoder (`vendor/qrcode.js`, public-domain `qrcode-generator` port) so nothing leaves the machine just to render a code.
27. **View on Sci-Hub button** + `scihub-fullscreen.js` content script — auto-expands the PDF viewer to fill the tab on any Sci-Hub mirror domain (generic "hide every sibling of the PDF embed" approach, since the actual Sci-Hub HTML couldn't be inspected — this sandbox's network can't reach `sci-hub.*`, confirmed via both plain `requests` and a headless-Chromium test with stealth patches, both blocked by Cloudflare).
28. **References panel** — popup button lists the current paper's references (Crossref `reference` array; bare-DOI entries get a secondary Crossref lookup for title/author, throttled 3-at-a-time with one retry to avoid 429s). Prefetched the moment a DOI is detected so the panel opens instantly. Whole row is clickable (opens on Sci-Hub).
29. **"Download All Works" + author page** (`author.html`/`author.js`) — originally asked for as a PhilPapers + Google Scholar scrape; both were dropped after live testing: Scholar has no free API and blocks scrapers, and PhilPapers sits behind a Cloudflare challenge that blocked both plain HTTP and a real headless-Chromium attempt. Rebuilt entirely on Crossref's public author-search API (`query.author`, up to `rows=1000`, filtered to exact family+given match) instead — legitimate, no blocking, returns DOIs directly. Features on the page:
    - Checklist of every matched work, batch-downloads selected ones sequentially into `<output folder>/<author name>/`.
    - Writes its own per-author `download_log.txt` (native host `append_log`/`read_log` actions) and skips already-downloaded DOIs on reopen; "Retry Failed" re-runs just the failed/corrupt ones.
    - Sort dropdown: relevance / newest / oldest / downloadable-first / by-journal / most-cited.
    - Filter-by-title search box (purely visual, doesn't touch underlying data/indices) + Cmd/Ctrl+F shortcut to focus it.
    - Progress bar + live ETA ("est. Nm Ns remaining") during a batch run, based on rolling average time-per-item.
    - "Open Folder in Finder" button (native host `open_folder` action), shown once anything's been downloaded.
    - Year-published line-chart histogram (SVG, `preserveAspectRatio="none"`) — click a point to filter the list to that year, shows the peak year, and warns if the year range is suspiciously wide (>90 years), which usually means Crossref's fuzzy name search caught more than one person.
    - Google Scholar + "Institutional Website" buttons — Scholar is a direct quoted-name search; Institution uses the author's affiliation when Crossref happens to have it on one of their papers, else falls back to a generic "`<name>` university faculty page" search.
    - Per-work citation count (Crossref `is-referenced-by-count`) shown in the metadata line.
30. **"Common Collaborators" page** (`collaborators.html`/`collaborators.js`) — same Crossref author-search data, tallies co-author frequency (excluding the target author) into a ranked list; each row links into that person's own "Download All Works" page.
31. **Removed on request:** a "Send to Device" button (Chrome has no API to trigger its native "Send to Your Devices" — tried `navigator.share()` as the closest substitute, then the user asked to drop it entirely) and a "Recheck Availability" popup button (built as part of the retry-count fix, removed after confirming the underlying fix alone was sufficient).

## Third build session — features added, roughly in order

32. **Fixed "More by This Author" / "Download All Works" / "Common Collaborators" not working on SAGE** — SAGE pages have zero `citation_author`/`author`/`dc.creator` meta tags for restricted-access articles (confirmed via live DevTools console check on the actual page, since Cloudflare blocks this sandbox's `curl`/headless-Chromium from ever seeing SAGE's real HTML). Two fixes in `content.js`'s `findMetadata()`:
    - Made the `dc.creator` meta-tag selectors case-insensitive (`meta[name="dc.creator" i]`) — SAGE actually emits `dc.Creator` (capital C), and CSS attribute selectors are case-sensitive by default, so the old selector silently missed it.
    - Added a DOM fallback that scrapes schema.org `Person` microdata (`[property="author"][typeof="Person"]` with `givenName`/`familyName` spans) for sites like SAGE that skip citation meta tags entirely but still mark up the byline.
33. **Year-published histogram gets real axis labels** — previously only showed values on hover; added an always-visible Y-axis (max/mid/0 count) and X-axis (auto-spaced year ticks) using plain positioned HTML/CSS overlaid on the chart, not SVG `<text>`, since the chart's `preserveAspectRatio="none"` stretches x/y independently and would visually distort real text.
34. **Full color-theme system** (`theme.js` + per-page CSS variable blocks) — started as a single light/dark toggle, then expanded on request into 5 selectable palettes: Dark (default), Warm Parchment, Cool Slate, Soft Sage, Pure Minimal. `popup.html` had zero CSS variables before this (100% hardcoded hex) and was fully converted. Settings page got a proper "Appearance" dropdown; the popup/author/collaborators/issue pages keep a quick ☀/🌙 icon toggle that flips between Dark and whichever light theme was last picked in Settings.
    - Several header-layout bugs surfaced and got fixed along the way: icons not vertically centered (`align-items: baseline` → `center`), the title wrapping to two lines and squishing the header (added `white-space: nowrap` + ellipsis, tightened letter-spacing), a stray empty gradient "logo" swatch removed from the Settings header, and the title text itself removed entirely from the popup header after repeated width complaints (popup header is now just the status dot + icon buttons).
35. **Institutional proxy access — added, then fully removed at user's request.** Built as an EZproxy-style URL rewriter (`journals.sagepub.com` → `journals-sagepub-com.<proxy-domain>`, confirmed against the user's actual `jmi.mapmyaccess.com` examples) with a Settings field and a popup button. Debugged one real bug in the process (the setting wasn't actually persisting — traced via a live `chrome.storage.sync.get` console check) before the user asked to remove the whole feature; fully reverted from `popup.html`/`popup.js`/`options.html`/`options.js`.
36. **"Download This Issue"** (`issue.html`/`issue.js`) — given a paper's DOI, looks up its volume/issue/ISSN via Crossref (`getIssueInfo`), then lists every article in that issue (`getIssueWorks`) with the same batch-download UI as the author-works page (per-issue output folder + log, skip-already-downloaded, retry-failed, progress bar).
    - Hit and fixed a real bug: Crossref's REST API has **no** `volume`/`issue` filter key (`filter=volume:...,issue:...` 400s) — switched to filtering by publication year (a real filter key) and matching volume/issue client-side against the results.
37. **Journal Watchlist** — subscribe to a journal from the "Watch This Journal" button on its issue page; a `chrome.alarms`-driven background check every 6 hours polls each watched journal's latest work on Crossref and notifies (clickable, opens the new issue's download page) when the volume/issue advances past the stored baseline. Settings page lists watched journals with a ✕ to unwatch and a manual "Check Now".
38. **Abstract hover preview** — author/issue/search list rows with a Crossref abstract (added `abstract` to the `select=` param on those queries, stripped from JATS XML to plain text) get a dotted-underline title; hovering shows the abstract in a floating tooltip that tracks the cursor.
39. **"Find Similar" / Similar Papers search** (`search.html`/`search.js`) — a "Find Similar" link next to any abstract-bearing title does crude keyword extraction (strips ~40 stopwords, takes the 6 most frequent remaining words ≥4 letters) and opens a new results page built on Crossref's `query.bibliographic` search, using the same download-list UI as the issue page. Results can chain into further similarity searches.
40. **Dark-mode-aware QR frame** — the QR code's actual modules stay pure black-on-white (a themed/inverted code risks failing stricter scanner apps, not worth the risk), but it's now wrapped in a themed card (`var(--bg-panel)`/`var(--border)`) instead of sitting as a stark white square directly on the popup background.
41. **"Grab DOI from selection"** context menu — select any DOI-looking text (or a `doi.org/...` link) anywhere on a page, right-click → grab it. Reuses the same `downloadDOI()` path as the `Alt+D` shortcut. A separate, earlier right-click "Grab DOI" attempt (item 10, first session) was reverted for reliability; this one is scoped to explicit user-selected text instead of trying to auto-detect from the click target, which is a meaningfully different approach.
42. **Author Watchlist** — same pattern as the Journal Watchlist but keyed on author name instead of ISSN; "Watch This Author" button lives on the author-works page header. Polls Crossref for that author's most recent work (name-matched the same way "Download All Works" matches) on the same 6-hour alarm, notifies on a new DOI, opens their works page on click.
43. **Fixed popup closing when clicking a reference** — `openSciHubPage`'s `chrome.tabs.create` was opening the new tab in the foreground, which steals focus and auto-closes the extension popup (standard Chrome popup behavior). Added `active: false` so reference-list clicks open a background tab and the popup stays open.

## Fourth build session — features added, roughly in order

44. **"Find Similar" now works even without a Crossref abstract** (`author.js`/`issue.js`/`search.js`) — previously gated entirely behind `work.abstract`, so most works (many publishers, especially SAGE, never deposit an abstract with Crossref) never showed the link at all. `openSimilarSearch` now falls back to `extractKeywords(work.abstract || work.title || "")`, and the "Find Similar" span in the meta row is rendered unconditionally instead of inside an `if (work.abstract)` block. The abstract-hover tooltip itself stays abstract-only (correctly — there's no text to show without one).
45. **Carrot color theme** — sixth palette added alongside Dark/Parchment/Slate/Sage/Minimal (cream background, carrot-orange accent, leafy-green "ok" color). Added to `theme.js`'s `LIGHT_THEMES` array and as a `:root[data-theme="carrot"]` CSS block in all 6 themed pages (`popup.html`, `options.html`, `author.html`, `collaborators.html`, `issue.html`, `search.html`), plus a dropdown entry in Settings.
46. **"Cited By" panel** (popup) — new button/panel next to References, same prefetch-on-DOI-detect → cache → instant-open pattern. Uses Semantic Scholar's free Graph API (`/graph/v1/paper/DOI:{doi}/citations`, no key required) since Crossref has no forward-citation index. New `getCitedBy` handler in `background.js`; `https://api.semanticscholar.org/*` added to `manifest.json` host permissions. Each row shows title/author/year and opens the citing paper on Sci-Hub via the existing `openSciHubPage` background-tab path.
    - **Bug fixed same session:** the first version requested `fields=...,authors.name,year`, but `authors.name` isn't valid dot-notation on this endpoint — Semantic Scholar rejected it with a 400 on literally every lookup (confirmed by reproducing directly against the live API, not paper-specific). Fixed to plain `authors` (returns full `{authorId, name}` objects per author).
    - Added a single retry with a 2s backoff on `429 Too Many Requests`, since Semantic Scholar's unauthenticated tier rate-limits aggressively — hit it firsthand while debugging, same pattern already used for Crossref's reference-title backfill.
47. **Bug fixes from a post-hoc audit** (subagent code review, not tied to a user-reported error):
    - `popup.html`'s light themes (parchment/slate/sage/minimal, and the new carrot) never overrode `--dot`/`--delete-bg`/`--delete-bg-hover`/`--delete-text`, so the toolbar status dot and "Delete Corrupt File" button silently stayed on the dark theme's color values in every light theme. Added the missing overrides to all 5.
    - The References-panel and Cited-By-panel "wait for prefetch to land" `setInterval` polls had no bail-out — if the prefetch errored without ever calling its `sendMessage` callback, or the popup navigated to a different DOI mid-poll, the interval would just poll forever (harmless in practice since Chrome tears down the whole popup document on close, but fragile). Both now clear themselves after 75 ticks (~15s) or on a DOI mismatch.

## Fifth build session — features added, roughly in order

48. **Configurable popup shortcuts** (`shortcuts.js`, new) — Chrome's `chrome.commands` API caps an extension at 4 assignable OS-level shortcuts, nowhere near enough for all ~12 popup buttons, so this is a separate, popup-only layer: single keypresses (d/c/l/v/a/w/o/i/q/r/b/g by default) that fire only while the popup has keyboard focus, listened for in `popup.js` and skipped when focus is in an input or a modifier key is held. `POPUP_SHORTCUT_ACTIONS` (in `shortcuts.js`, shared by `popup.js` and `options.js`) is the single source of truth for the action list and defaults. Settings gets a new "Popup Shortcuts" card — click "Change" on any row, press a key, it saves to `chrome.storage.sync`'s `popupShortcuts` map; assigning a key already in use frees it from whichever action held it first, so nothing double-fires. The existing Alt+D/Alt+F `chrome.commands` shortcuts are untouched and still work page-wide, not just when the popup is open.
49. **Mirror Health reset** — new `reset_mirror_health` native-host action (`doi_host.py`) that pops one mirror's entry (by URL) or clears the whole `mirror_health.json` file. Settings' Mirror Health panel gets a per-row "Reset" button plus a "Reset All" button next to Refresh.
50. **Mirror latency sparkline** — the mirror race in `scihub_download.py` now keeps a rolling 20-entry `latency_history` list per mirror (trimmed with `del history[:-20]`), exposed through `doi_host.py`'s `mirror_health` action and rendered as a small inline SVG polyline (`buildSparkline()` in `options.js`) next to each mirror's status text; hovering shows the min–max range.
51. **Theme-aware toolbar badge** — the ✓/✗ badge color was hardcoded to the dark theme's green/red regardless of the active color theme. `background.js` now has a `THEME_BADGE_COLORS` map (mirroring each theme's `--ok`/`--err` CSS values, since a service worker has no DOM to read the real ones from) and reads the current `theme` from `chrome.storage.sync` on every badge update rather than caching once at service-worker startup, so it stays correct across theme switches without a restart.
52. **Batch Download Multiple Issues** (`issue.html`/`issue.js`) — new card: given a volume range (max 50) and an issue number (defaults to the current issue), downloads each volume sequentially into its own subfolder + log, reusing the single-issue page's naming/skip-already-downloaded conventions.
53. **Pause/Cancel for batch downloads** — refactored the volume-range batch into a shared `downloadIssueGroup()` helper plus a `batchControl = { paused, cancelled }` state object, checked between every single DOI (not just between issues) so a click takes effect within seconds. A shared Pause/Resume + Cancel row appears whenever any batch mode is running. This same infrastructure was then reused, unmodified, by every batch feature added afterward.
54. **Download Entire Journal** — new `getAllJournalWorks` handler in `background.js`, deep-paginating Crossref's cursor API (up to ~20k works across 20 pages, 300ms between pages to stay polite) for a whole ISSN, since Crossref has no "list all issues" endpoint. Results are grouped by volume/issue client-side, sorted, and walked through the same `downloadIssueGroup()`/pause/cancel machinery as the volume-range batch. Gated behind a `confirm()` warning since it can run for a very long time and download a lot of files.
55. **Search This Journal** (`issue.html`/`issue.js`) — keyword search scoped to the current ISSN via a new `searchJournalKeyword` handler in `background.js`, using Crossref's `filter=issn:<issn>&query=<keyword>` (a real, targeted server-side search — cheap, unlike walking the whole journal). Each result has its own "Download" button; a "Download All Matches" button runs the downloadable subset through the shared batch/pause/cancel infrastructure, filed under a `<journal> Keyword <query>` subfolder.
56. **Keyword highlighting** — new `escapeHtml()`/`highlightTerms()` helpers in `issue.js` wrap matched search terms in `<mark>` (styled with the active theme's accent color) in both result titles and the abstract hover tooltip. `showAbstractTooltip()` gained an optional `highlightQuery` param for this.
57. **Retry Failed for keyword-search batches** — `downloadIssueGroup()` now also returns `failedWorks` (the actual work objects, not just a count) so a "Retry Failed (N)" button can re-run just the failures, matching the pattern already used on the issue/author pages.
58. **Sort search results** — Relevance / Most Cited / Newest / Oldest dropdown, mirroring the existing `sortWorks()` pattern from the main issue list. `searchJournalKeyword` now also selects `published-print`/`published-online` and extracts `year` (previously only volume/issue/citations). `lastSearchMatches` stays in Crossref's original relevance order; sorting renders from a sorted copy via `sortSearchMatches()`.
59. **Search result pagination ("Load More")** — `searchJournalKeyword` takes an `offset` param and returns Crossref's `total-results` count; a "Load More" button fetches the next 50 and appends them, re-rendering through whatever sort is active. Status line shows e.g. "50 matches found of 287 total" when more exist.
60. **"↑ Jump to Top"** — small button next to Load More, shown once results exceed 20 rows, smooth-scrolls back to the Search This Journal card.
61. **"Expand All Abstracts"** — toggle button that switches every result row from hover-only abstract tooltips to an always-visible abstract block underneath the title (with the same keyword highlighting), for skimming many results without hovering each one individually. Hover listeners are skipped on rows when this is on, so nothing double-shows.

## Sixth build session — features and fixes, roughly in order

62. **Fixed literal `&amp;` showing in journal names/titles** — Crossref bakes HTML entities directly into its metadata strings (e.g. `container-title` for "Philosophy & Social Criticism" is literally `"Philosophy &amp; Social Criticism"`, not markup), so setting it via `textContent` displayed the raw escape sequence. Added `decodeHtmlEntities()` in `background.js` (handles named + numeric entities by hand, since a service worker has no DOM/DOMParser to lean on) and applied it at every point Crossref data is extracted — titles, journal/container-title, author names, reference titles, Paper of the Day title — across every handler. Since all Crossref parsing is centralized in `background.js`, this one fix propagated to every page automatically.
63. **"Download Entire Journal" now opens a dedicated tab** (`journal-download.html`/`journal-download.js`, new files) — walking hundreds of issues and thousands of works was too much for the shared `#batch-status-list`'s single-line-per-issue format, so this superseded that inline approach entirely. The new page: fetches the whole journal once (`getAllJournalWorks`), groups by volume/issue, and renders each issue as a native `<details>` foldable card with every article listed inside showing live per-row status (Pending → Downloading… → Downloaded ✓/Failed/Corrupt). A sticky toolbar at the top holds Pause/Resume + Cancel (own standalone `batchControl` state, not the shared one in `issue.js`) plus a progress bar and a persistent "N issues found — M articles total" line, so both stay visible while scrolling a long list. `issue.js`'s old `runWholeJournalBatch()`/`groupWorksByIssue()` were deleted; the button now just opens this tab after the same `confirm()` warning.
64. **Per-article "remove before download"** (`journal-download.js`) — each Pending row gets a small ✕ button; clicking it marks `work.removed = true`, greys/strikes the row, and the download loop skips it entirely (no `sendDOI` call, not counted toward done/failed). Decrements the running total denominator live and updates that issue's still-collapsed "N works" count. ## Known quirks / context for next time

- `scihub_download.py` hardcodes its log path and mirror-health path to `/Users/floppa/doi-extension/native-host/` — fine as-is since it's a single-machine personal tool, but not portable without editing those constants.
- The extension is loaded unpacked from `~/Downloads/doi-extension/extension`; the native host lives at `~/doi-extension/native-host` (moved out of Downloads specifically to dodge Gatekeeper quarantine issues). If either path changes, the native messaging manifest (`com.doi_grabber.host.json` in Chrome's `NativeMessagingHosts` directory) needs updating too.
- There's a separate, independent auto-renaming tool (`~/Scripts/auto_rename_by_doi.py`, Folder Action-triggered) that renames files in the output folder after the fact — this extension's "Reveal in Finder" accounts for it, but any future feature touching the output folder should keep it in mind.
- User pushed back repeatedly on open-ended "give me more feature ideas" requests without picking any — turned out to be idle brainstorming with no intent to build; keep future batches short and light rather than fully fleshed out, and don't ask "why" again unless the pattern changes.
- Crossref's `query.author` is a relevance-ranked full-text search, not a guaranteed-complete author filter — even at the `rows=1000` cap, an extremely prolific author's more obscure works could in theory rank beyond that window and never be fetched. No exact "everything by this person" endpoint exists without an ORCID.
- PhilPapers and Google Scholar are both permanently off the table for scraping (Cloudflare + no free API, respectively, both confirmed by direct testing) — any future "more sources for author works" ask should go back to Crossref refinements (or an ORCID lookup) rather than retrying either of those.
- `author.html`/`author.js`/`collaborators.html`/`collaborators.js` and `vendor/qrcode.js` are new files not mentioned in the original Architecture section above — same folder (`extension/`), same load-unpacked setup.
- Crossref's REST API filter list does **not** include `volume` or `issue` as valid filter keys — `filter=volume:X,issue:Y` 400s. Anything that needs to narrow to a specific issue has to filter by a real key (e.g. `from-pub-date`/`until-pub-date`) and then match volume/issue client-side against the results, as done in `getIssueWorks`.
- The Sci-Hub-piracy core of this tool (not the Crossref-metadata parts) means it's a hard no for Chrome Web Store distribution — that's explicitly against store policy and would get rejected/pulled regardless of code quality. GitHub distribution is viable policy-wise, but the Native Messaging host (`doi_host.py` + `scihub_download.py` + the registered native-messaging manifest) is a real per-machine setup burden for anyone else trying to use it — not something a repo README alone fully solves. Discussed but not acted on.
- Only *ever* create Chrome extension context menus inside `chrome.runtime.onInstalled` (not `onStartup`) — the browser persists menu items itself across service-worker restarts, so calling `chrome.contextMenus.create()` again with the same `id` on every startup throws a duplicate-id error. `chrome.alarms.create()`, by contrast, *is* safe to call repeatedly (idempotent — just resets the alarm), so those are recreated in both `onInstalled` and `onStartup` as a startup-safety net.
- A Chrome extension popup auto-closes whenever a newly opened tab takes focus — any `chrome.tabs.create({ url })` triggered from inside the popup (e.g. clicking a references-list row) needs `active: false` or the popup vanishes out from under the user.
- The journal watchlist and author watchlist share one 6-hour `chrome.alarms` cadence (`checkWatchlist`/`checkAuthorWatchlist`, both fired from the same `onAlarm` listener) and one generalized notification-click router (`pendingWatchNotifications[notifId] = { page, ...params }`, where `page` is `"issue.html"` or `"author.html"`) — any future third watchlist type (e.g. "watch this publisher") should follow the same `page`-tagged pattern rather than inventing a new click-routing mechanism.
- The "Watch This Journal" / "Watch This Author" buttons deliberately do **not** live in the popup — they were built there first, then moved to the issue/author pages respectively after the user found the popup too cluttered. Any future per-entity toggle (watch/star/etc.) should probably default to living on that entity's dedicated page, not the popup.
- Semantic Scholar's Graph API `fields=` parameter is picky about dot-notation: on the `/paper/{id}/citations` endpoint, nested fields like `authors.name` are rejected outright (400), but the plain top-level field name (`authors`) works and returns the full object. Test any new `fields=` combination against the live API before shipping — a bad field name breaks the endpoint for every request, not just edge cases.
- Semantic Scholar's unauthenticated tier rate-limits hard and fast — hit a 429 within a handful of manual test calls. Any future feature hitting that API should assume it needs the same throttle/retry treatment already used for Crossref reference backfills, not a bare `fetch`.
- When adding a new color theme (or any per-theme CSS var), check it against **all 6** themed HTML files (`popup.html`, `options.html`, `author.html`, `collaborators.html`, `issue.html`, `search.html`) — `popup.html` in particular has extra vars (`--dot`, `--delete-bg`, `--delete-bg-hover`, `--delete-text`) that the other 5 files don't, and it's easy to copy a theme block from one of the simpler pages and silently miss them.
- Chrome caps `chrome.commands` at 4 assignable OS-level shortcuts total (including `_execute_action`) — there is no way to give every popup button its own global hotkey through that API. Any future "more shortcuts" ask should extend the existing popup-only `shortcuts.js`/`POPUP_SHORTCUT_ACTIONS` layer (keypresses while the popup has focus), not try to add more `commands` entries to the manifest.
- Whenever `doi_host.py` or `scihub_download.py` changes, reloading the unpacked extension at `chrome://extensions` is **not enough** — the native host is a separate long-lived process Chrome only re-spawns on a fresh connection after a full Chrome restart (or disable/re-enable the extension). Forgetting this makes a native-host change look like it silently didn't take effect.
- `mirror_health.json` entries now carry a `latency_history` array (rolling last 20, trimmed via `del history[:-20]`) alongside the existing `fail_count`/`last_latency_ms`/`last_seen` fields — any future code reading/writing that file directly (rather than through `doi_host.py`'s `mirror_health`/`reset_mirror_health` actions) needs to preserve or explicitly reset that field too.
- `mirror_health.json` entries also carry `latency_by_hour` (a `{"0"–"23": [up to 10 latencies]}` dict, keyed by hour-of-day string) feeding the seed-ordering in item 70 — same rule as `latency_history` above: preserve or explicitly reset it if writing the file directly. Existing entries created before this feature simply lack the key until their mirror is raced again during that hour; `hourly_avg_latency()` treats a missing/empty bucket as "no data" rather than erroring.
- The batch-download infrastructure (`startBatchControls()`/`endBatchControls()`/`downloadIssueGroup()`/`waitWhilePaused()` in `issue.js`) is shared and singleton — only volume-range and keyword-search batches use it now (whole-journal was moved out to its own tab, see below); the Pause/Cancel row and `#batch-status-list` are shared DOM across those two. Any future batch-shaped feature *on `issue.html` itself* should plug into that same infrastructure rather than building parallel pause/cancel/status-list handling.
- Crossref's `filter=issn:<issn>` **is** a valid filter key (unlike `volume`/`issue`), so keyword search (`searchJournalKeyword`) can do a single targeted server-side query instead of the deep-pagination-then-filter-client-side approach `getAllJournalWorks`/`getIssueWorks` are forced into.
- `journal-download.html`/`journal-download.js` is a separate tab/document context from `issue.html` — it does **not** reuse `issue.js`'s shared batch infrastructure (can't share JS state across tabs without messaging) and instead has its own standalone `batchControl`/`sanitizeFolderName`/`getSettings`/`groupWorksByIssue` copies. Any future change to the shared batch pattern in `issue.js` (e.g. a new pause/cancel behavior) needs to be mirrored here by hand if it should apply to whole-journal downloads too.
- Crossref's metadata frequently contains literal HTML entities baked into the text itself (`&amp;`, etc.) rather than real markup — anything that pulls a new field out of a Crossref response (title, journal, author name, abstract, anything else) should run it through `decodeHtmlEntities()` in `background.js` before returning it, or the raw escape sequence will show up on-screen.
- Color emoji (🌙, ☀, etc.) used as button icons render visibly larger than plain text at the *same* `font-size` — don't assume matching font-size against adjacent button text will look visually balanced; check it renders like `popup.html`'s icons (13px) rather than trusting the number alone.
- The skeleton-loading shimmer CSS (`.skeleton-row`/`.skeleton-bar`/`@keyframes skeleton-pulse`) is duplicated per-file rather than shared, matching this codebase's existing pattern of repeating theme CSS blocks across pages instead of a common stylesheet — when adding a new loading state, copy the pattern from whichever page's `--surface`/`--surface-2`/`--border` naming matches (`popup.html` uses different var names, `--bg-panel`/`--btn-bg`, than the rest).
- `chrome.storage.local` (not `.sync`) now holds `potdNonce` (per-day nonce for "Paper of the Day"'s deterministic pick) and `potdHistory` (capped 50-entry array) — deliberately local rather than synced across devices, since this is per-machine download history anyway (the download log itself is local-only too).
- Crossref's abstract coverage is genuinely spotty (confirmed directly against the live API for several DOIs) — a missing abstract there isn't a parsing bug, it's often just not submitted by the publisher. OpenAlex (`api.openalex.org`) has meaningfully better coverage for the same works and is now the first-tried source (`getWorkAbstract` in `background.js`); Crossref is only the fallback. OpenAlex abstracts come back as a word→positions inverted index, not plain text — see `reconstructOpenAlexAbstract()`.
- OpenAlex's abstract field is not always trustworthy — it mis-scraped endnotes-as-abstract for some Taylor & Francis review-essay/notes pieces (always prefixed with Tandfonline's "Click to increase image size..." figure-viewer boilerplate). `reconstructOpenAlexAbstract()` filters that pattern out; if a future publisher shows the same failure mode with different boilerplate text, extend that filter rather than assuming OpenAlex data is always clean.
- Tandfonline (and likely other Cloudflare-protected publishers) returns a 403 bot-challenge page to any plain `requests`/`fetch` call — confirmed for both abstract-scraping and PDF-scraping attempts. Nothing server-side (native host or background service worker) can get past this; it would need a real browser tab executing the challenge JS. Don't re-attempt a bare-HTTP scrape against Tandfonline expecting a different result — it's a structural block, not a flaky one.
- Unpaywall's `oa_locations`/`best_oa_location` entries can have `url_for_pdf: null` with a `url` field that's just the bare `doi.org` resolver link, not a real PDF — `get_oa_pdf_url_unpaywall()` in `scihub_download.py` only trusts `url_for_pdf` for exactly this reason. Any future code reading Unpaywall data directly should do the same rather than assuming `url` is always a usable fallback.
- `scihub_download.py`'s Sci-Hub → Unpaywall → publisher-page fallback chain only engages when Sci-Hub returns *zero* candidate URL, not when a Sci-Hub download comes back corrupt (fails the `%PDF-` check) — a corrupt Sci-Hub result does not currently retry through the OA tiers. Worth revisiting if corrupt-from-Sci-Hub turns out to be common enough to matter.
